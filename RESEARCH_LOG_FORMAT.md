# รูปแบบบันทึกการวิจัยของ Lily

เอกสารนี้กำหนดรูปแบบบังคับสำหรับไฟล์ใน `research_log/` บันทึกเหล่านี้มีหน้าที่เล่างานวิจัยให้มนุษย์อ่านเข้าใจได้โดยไม่ต้องเปิดไฟล์ JSON หรืออ่าน source code ก่อน

บันทึกการวิจัยไม่ใช่การนำผลจาก script มาวางต่อกัน แต่ต้องเล่าเป็นลำดับว่าเราพบปัญหาอะไร ตั้งสมมติฐานอย่างไร ทดลองอย่างไร พบอะไร ผลนั้นหมายความว่าอย่างไร มีข้อจำกัดอะไร และควรเดินต่อทางไหน

## หลักภาษา

- เขียนด้วยภาษาไทยธรรมดา ประโยคสั้น และอธิบายความหมายของตัวเลขสำคัญ
- ใช้ภาษาอังกฤษเฉพาะชื่อเฉพาะ คำสั่ง path ตัวแปร และศัพท์เทคนิค เช่น `backtest`, `MinTRL`, `PSR`, `DSR`, `E1`, `scope_restricted`
- หากใช้ศัพท์เทคนิคที่ผู้อ่านอาจไม่คุ้น ให้อธิบายเป็นภาษาไทยในครั้งแรก
- ห้ามมีข้อความเพี้ยน ตัวอักษรทดแทน หรือ mojibake
- ถ้าหลักฐานไม่พอ ให้เขียนตรง ๆ ว่า `ยังสรุปไม่ได้` พร้อมบอกว่าขาดอะไร
- แยกข้อเท็จจริง การตีความ และข้อเสนอแนะออกจากกัน ห้ามทำให้ผลดูดีกว่าหลักฐาน

## การตั้งปัญหาและคำถามวิจัย

คำถามวิจัยต้องสั้น ชัด และมีขอบเขต ไม่ใช้คำถามลอย เช่น “Trend following ดีไหม” หรือ “ควรศึกษาอะไรต่อ”

คำถามที่ใช้ได้ต้องบอกให้ผู้อ่านรู้ว่า:

1. กำลังตรวจสิ่งใดหรือกลุ่มใด
2. ใช้เงื่อนไข ช่วงเวลา หรือขอบเขตใด
3. วัดผลลัพธ์อะไร
4. อะไรคือเกณฑ์ที่ทำให้ตอบว่า ผ่าน ไม่ผ่าน หรือยังสรุปไม่ได้

ตัวอย่างที่ชัด:

> ภายใต้ต้นทุนและข้อจำกัดที่ล็อกไว้ พอร์ต ETF แบบ 60-day directional count ให้ผลตอบแทนสุทธิเป็นบวกในช่วง falsification และมีจำนวน independent-bet equivalents ถึงเกณฑ์หรือไม่

ทุกบันทึกต้องมีสี่บรรทัดนี้ในส่วนที่ 2:

```markdown
- คำถามวิจัย: <คำถามหนึ่งประโยค ไม่เกิน 240 ตัวอักษร>
- ขอบเขต: <ข้อมูล กลุ่มตัวอย่าง ช่วงเวลา และสิ่งที่ไม่รวม>
- สมมติฐาน: <สิ่งที่คาดว่าจะเกิดและเหตุผล>
- เกณฑ์ตัดสิน: <เงื่อนไขผ่าน ไม่ผ่าน หรือยังสรุปไม่ได้>
```

## การตั้งชื่อและลำดับ

ใช้รูปแบบ:

```text
NNN-lily-<clear-topic-slug>.md
```

- `NNN` เป็นเลขสามหลักต่อเนื่อง เริ่มจาก `001`
- `lily` เป็นคำคงที่
- slug ต้องอ่านแล้วรู้หัวข้อ เช่น `l0-sizing-feasibility` หรือ `l1-baseline`
- การแก้ความถูกต้องของการทดลองเดิมให้แก้เลขเดิม การทดลองรอบใหม่จริงจึงใช้เลขถัดไป

Timestamp ใช้ UTC รูปแบบ `YYYY-MM-DDThh:mm:ssZ`

## โครงสร้างบังคับ 6 ส่วน

```markdown
# บันทึกการวิจัย NNN: <ชื่อเรื่องภาษาไทยที่ชัดเจน>

## 1. ข้อมูลพื้นฐาน

- Timestamp UTC: `<YYYY-MM-DDThh:mm:ssZ>`
- โครงการ: Trend Following - Lily
- Hypothesis ID: `<L-x>`
- Experiment ID: `<รหัสการทดลอง>`
- ผู้บันทึก: `<ชื่อ>`
- ระดับหลักฐาน: `<E0/E1/E2/E3>`
- ข้อสรุป: `<ผ่าน/ไม่ผ่าน/ยังสรุปไม่ได้/จำกัดขอบเขต>`
- Artifact หลัก: `<repo-relative path>`
- Producing commit: `<40-character git hash>`

### อ่านแบบเร็ว

<สรุป 2-5 ประโยคว่าเราถามอะไร พบอะไร ผลหมายความว่าอย่างไร และยังห้ามสรุปอะไร>

## 2. ปัญหา (คำถาม) และสมมติฐาน

- คำถามวิจัย: <คำถามหนึ่งประโยค>
- ขอบเขต: <ขอบเขตที่วัดและสิ่งที่ไม่รวม>
- สมมติฐาน: <ผลที่คาดและกลไก>
- เกณฑ์ตัดสิน: <กติกาตัดสินที่ล็อกก่อนเห็นผล>

<อธิบายว่าทำไมคำถามนี้สำคัญและเชื่อมกับสมมติฐานหลักอย่างไร>

## 3. ขั้นตอนการทดลอง

1. <ข้อมูลและ provenance>
2. <กติกาที่ล็อกก่อนวัด>
3. <วิธีคำนวณหรือเปรียบเทียบ>
4. <validator และ guardrail>

## 4. ผลลัพธ์

<รายงานทั้งผลที่ผ่านและไม่ผ่าน พร้อมตัวเลขสำคัญ>

<อธิบายความหมายของตัวเลขด้วยภาษาไทย ห้ามปล่อยให้ตารางพูดแทนทั้งหมด>

## 5. อภิปรายผล ปัญหา และข้อจำกัด

<แยกสิ่งที่ข้อมูลสนับสนุนออกจากการตีความ อธิบายความเสี่ยงที่ผลอาจหลอกเรา>

สิ่งที่ห้ามสรุปจากการทดลองนี้:

- <ข้อห้าม 1>
- <ข้อห้าม 2>

## 6. สรุปผลการทดลองและแนวทางพัฒนาต่อ

ข้อสรุป: `<ผ่าน/ไม่ผ่าน/ยังสรุปไม่ได้/จำกัดขอบเขต>` <ประโยคสรุปที่ตอบคำถามวิจัยโดยตรง>

แนวทางพัฒนาต่อ:

1. <ขั้นตอนถัดไปที่หลักฐานรองรับ>
2. <หากล้มเหลว ระบุสมมติฐานใหม่หรือคำถามใหม่ที่ควรตรวจ>
```

## เมื่อใดต้องมีบันทึก

ต้องเขียนหรือแก้บันทึกเมื่อ:

- จบการทดลองหนึ่งรอบ
- จบ diagnostic ที่เปลี่ยนความเข้าใจหรือการตัดสินใจ
- falsify, park, จำกัดขอบเขต หรือยกเลิกสมมติฐาน
- พบว่าคำถามเดิมตั้งผิดและต้องตั้งคำถามใหม่
- แก้ผลเดิมเพราะพบปัญหาด้านข้อมูล สถิติ หรือ implementation

งานโครงสร้างที่ไม่มีผลเชิงวิจัยไม่ต้องสร้างบันทึกใหม่ เว้นแต่เปลี่ยนกติกาการตีความหลักฐาน

## ความสัมพันธ์กับหลักฐานทางเครื่อง

- JSON report, search log, dataset registry และ validator เป็นหลักฐานละเอียดสำหรับตรวจซ้ำ
- Markdown ใน `research_log/` เป็นคำอธิบายสำหรับมนุษย์ แต่ตัวเลขทุกค่าต้องย้อนกลับไปยัง artifact ที่ระบุได้
- บันทึกไม่สามารถเปลี่ยนผลของ locked gate หรือยกระดับ evidence tier ด้วยถ้อยคำ
- รายงานการทดลองที่กำหนดไว้ใน `config/research_log_requirements.json` จะถือว่าไม่ครบถ้าไม่มีบันทึกที่ผ่าน audit

รูปแบบนี้ดัดแปลงจาก Higanbana `RESEARCH_LOG_FORMAT.md` แต่เปลี่ยนหัวข้อให้ตรงกับกระบวนการวิจัยของ Lily และเพิ่มข้อบังคับเรื่องคำถามที่สั้น ชัด มีขอบเขต และมีเกณฑ์ตัดสิน
